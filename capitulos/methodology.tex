\chapter{Metodologia}
Levando em consideração as barreiras enfrentadas por novatos em software livre, em especial aquelas relacionadas a documentação dos projetos, propomos por meio desta metodologia um modelo de classificação que contribua com a identificação de informações em documentações relevantes a novos contribuidores. A metodologia proposta dá continuidade a um trabalho preliminar \citep{fronchetti2019attracts}, onde foram investigadas características que influenciam a atratividade de novatos em projetos de software livre. 

A elaboração deste método é baseada em dois estudos relacionados, um deles proposto por \citet{steinmacher2014preliminary}, que evidenciam a necessidade de uma ferramenta que auxilie novatos na compreensão de documentações de projetos em de software livre, e outro proposto por \citet{panichella2015how}, que desenvolvem um método de identificação de seções em documentações de software, similar ao apresentado abaixo para este estudo. Este capítulo é divido em duas seções: Na Seção \ref{sec:questoes-de-pesquisa} são apresentadas e discutidas as questões de pesquisa deste estudo, e na Seção \ref{sec:metodo} é apresentado o método a ser executado.

\section{Método}
\label{sec:metodo}
Para que seja possível responder as questões desta pesquisa, o método proposto é dividido em um conjunto de cinco etapas, sendo que as três primeiras etapas já foram executadas preliminarmente. A primeira etapa consistiu em definir quais informações relevantes a novos contribuidores deveriam ser identificadas nas documentações de projetos de software livre. Para isso, um conjunto de categorias associados a problemas que novatos enfrentam foi estabelecido. A segunda etapa se concentrou na seleção dos projetos a serem utilizados como amostra para identificação de tais categorias. Neste  passo, a extração das documentações para os projetos selecionados também foi executada. Na terceira etapa, um estudo empírico foi proposto com novatos em software livre. O estudo empírico serviu para popular uma base de dados com análises das documentações extraídas, feitas com base nas categorias estabelecidas na etapa anterior. Para o futuro, um modelo de classificação será elaborado e avaliado a partir das análises coletadas durante o estudo empírico, e uma avaliação dos resultados do modelo de classificação a ser construído será realizada, utilizando como artifício de avaliação métodos estatísticos, e questionários a serem aplicados a novos e experientes contribuidores em software livre. Na Figura \ref{fig:metodo-pesquisa}, é apresentada a sequência de atividades referentes ao método desta pesquisa. Apesar das três primeiras etapas já terem sido executadas preliminarmente, é importante ressaltar que todas elas estão abertas a reavaliação.

\subsection{Categorias}
\label{sec:categorias}
Para identificar trechos relevantes a novos contribuidores nas documentações de projetos de software livre, um conjunto de categorias foram elaboradas de modo que fosse possível realizar a identificação. Estas categorias foram estabelecidas com base nas barreiras apresentadas pelo modelo de \citet{steinmacher2016overcoming}, e em demais trabalhos relacionados, que também identificaram dificuldades que novatos enfrentam neste contexto. As principais dificuldades enfatizadas nas categorias estabelecidas, foram aquelas passíveis de solução por meio da documentação dos projetos. Estas foram utilizadas durante a execução do estudo empírico (Seção \ref{sec:estudo-empirico}). \subsection{Projetos}
\label{sec:projetos}
Considerando que a premissa principal deste estudo é conseguir identificar trechos em documentações de projetos de software livre, um conjunto de projetos neste contexto foi selecionado como amostra. Os projetos foram escolhidos a partir de um estudo proposto por \citet{avelino2015truck}, que classifica projetos populares em software livre de acordo com seu risco de sobrevivência, utilizando o fator ônibus\footnote{O fator ônibus (do inglês, \textit{truck factor}), é uma medida de risco que investiga o quanto um determinado projeto de software depende de um número específico de pessoas. De modo geral, esse fator define quantas pessoas em um projeto precisam ser  ``atingidas por um ônibus'' para que o projeto fracasse ou se torne inativo.} dos projetos como métrica de avaliação. A justificativa para utilização dos projetos deste artigo está no fato de que todos contém alto índice de popularidade, estão divididos em seis linguagens de programação bem estabelecidas (JavaScript, Python, Ruby, C/C++, Java e PHP), e contém um número considerável de desenvolvedores ativos trabalhando em uma diversidade de arquivos por repositório. Como a ideia é contribuir principalmente com comunidades que necessitam de novos contribuidores, projetos que continham fator ônibus menor ou igual a três foram selecionados, gerando um total de noventa e seis projetos. 

De cada um dos projetos, foram extraídos os arquivos README e CONTRIBUTING dos repositórios de código fonte. A razão para escolha destes arquivos, e não de outras fontes de documentação, se concentra no fato de que ambos apresentam informações relevantes a novos contribuidores, são de fácil extração, e são escritos a partir de uma mesma sintaxe, o que facilita o processo de identificação automática das categorias. Visto que os projetos selecionados estavam hospedados na plataforma de codificação GitHub, as documentações acabaram sendo extraídas por meio da GitHub API\footnote{\url{developer.github.com}}, um serviço que fornece dados sobre repositórios de código hospedados nesta plataforma. Considerando que o objetivo desta pesquisa se concentra em investigar tanto os arquivos README como CONTRIBUTING, projetos que não continham ambos os arquivos, ou que continham os arquivos porém  estes estavam vazios, foram removidos da lista de seleção. Ao final da filtragem, restaram cinquenta e oito projetos, os quais foram selecionados para uso nas etapas posteriores. \subsection{Estudo empírico}
\label{sec:estudo-empirico}
Durante a revisão dos trabalhos relacionados, não foram constatados estudos ou ferramentas que identificassem categorias relevantes a novatos em documentações de projetos de software livre. Por esta razão, um processo manual de identificação das categorias estabelecidas foi elaborado. Para realização do processo de identificação das categorias, novatos almejando contribuir com projetos de software livre foram convidados a identificarem trechos nas documentações dos projetos selecionados anteriormente, que fossem relevantes a eles, utilizando como base as categorias estabelecidas nesta pesquisa. A produção do estudo empírico foi dividida em três partes: elaboração, seleção dos participantes e execução.

\subsubsection{Elaboração do estudo}
O processo de elaboração do estudo foi realizado em duas etapas:
\begin{itemize}
    \item \textbf{Organização dos documentos extraídos:}\\
    Para facilitar a execução do estudo empírico, os documentos README e CONTRIBUTING dos projetos selecionados foram organizados em um conjunto de planilhas. Cada planilha, contendo duas páginas, uma para cada arquivo de documentação, representava um projeto de software livre. As páginas de cada planilha, foram organizadas em colunas. Na primeira coluna, se encontravam os parágrafos do respectivo arquivo de documentação que a página representava. Nas colunas seguintes, foram dedicados espaços separados para identificação das categorias relevantes a novatos, que foram marcados pelos participantes durante a execução do estudo empírico. A Tabela \ref{tab:planilha} apresenta um exemplo de como as páginas de cada planilha representando os projetos selecionados foram organizadas.  
    
    \item \textbf{Produção do tutorial:}\\ 
    Para que os participantes do estudo tivessem entendimento acerca do que devia ser feito, um tutorial foi preparado. Este tutorial continha, ente outras coisas, a descrição das categorias estabelecidas na Seção \ref{sec:categorias}, a serem identificadas por eles durante o estudo, bem como os passos que cada participante deveria seguir durante o processo de identificação destas categorias nas documentações dos projetos. Todo o material tutorial produzido está disponibilizado publicamente online\footnote{\url{fronchetti.com.br/usp/experimento}}.
\end{itemize}

\subsubsection{Seleção dos participantes}
Tendo em vista que o alvo de contribuição deste estudo eram novatos em software livre, novos contribuidores foram convidados a participar da identificação das categorias. Os novatos convidados foram alunos de ensino superior, inscritos em disciplinas de desenvolvimento de software livre, que tinham como objetivo nas disciplinas contribuir para projetos neste contexto. Para que fosse possível aplicar o estudo empírico, os responsáveis por esta pesquisa entraram em contato com os professores de instituições de ensino superior, e solicitaram a aplicação do experimento com os alunos das disciplinas de software livre. Assim que a solicitação foi atendida por um grupo de professores, todos os trâmites legais, incluindo a avaliação do comitê de ética das universidades, foram realizados antes da execução do estudo empírico.

\subsubsection{Execução do estudo}
O estudo proposto foi executado em duas instituições de ensino, na Universidade de São Paulo (USP) e na Universidade Federal do Pará (UFPA). Ambas as execuções aconteceram durante as disciplinas de Desenvolvimento de Software Livre. A primeira versão do experimento foi aplicada para os alunos da UFPA. Por conta do número de computadores disponíveis na instituição, os alunos foram divididos em um total de quarenta e três duplas, sendo que cada dupla teve aproximadamente duas horas em sala de aula para analisar um único projeto de software livre. Todas as duplas terminaram o experimento no prazo determinado, resultando em quarenta e três planilhas analisadas. Com auxílio do professor responsável pela disciplina, dificuldades em relação ao experimento foram constatadas, e modificações foram feitas na elaboração do estudo, a fim de aprimorar a execução do experimento.

Para os alunos da USP, o experimento foi conduzido durante uma semana em que os alunos não tinham aula (popularmente conhecida na instituição como semana de \textit{break}). Com auxílio do professor responsável pela disciplina de desenvolvimento em software livre, os alunos foram instruídos uma semana antes da semana de \textit{break} sobre como o experimento deveria ser realizado em casa. O tutorial e o contato dos pesquisadores responsáveis pela pesquisa foram disponibilizados para os alunos, caso eles tivessem alguma dúvida. Nesta segunda versão do experimento, cada aluno ficou responsável por identificar as categorias em cinco projetos de software livre distintos, sendo que quinze alunos participaram do experimento, totalizando setenta e cinco planilhas analisadas. Todos foram convidados a submeter críticas ou sugestões para novas versões do experimento, porém nenhuma resposta quanto a isso foi recebida.

Além da planilha, o professor da USP também solicitou aos alunos que descrevessem as principais dificuldades enfrentadas durante a contribuição com projetos de software livre em sua disciplina, e diários contendo as dificuldades que estes enfrentaram foram compartilhadas junto as planilhas aos pesquisadores responsáveis por esta pesquisa, para eventual análise. Ao final da execução, cento e dezoito planilhas foram analisados por alunos de ambas as universidades. As seções a seguir, apresentam as duas últimas etapas ainda não executadas para neste método. 

\subsection{Modelo de classificação}
\label{sec:classificacao}
Para que seja possível automatizar o processo de identificação das categorias em documentações de projetos de software livre, um modelo de classificação será proposto como parte resultante desta pesquisa. A elaboração deste modelo acontecerá em três etapas: Na primeira etapa, as planilhas analisadas pelos alunos durante o estudo empírico serão convertidas em valores que possam ser interpretados por algoritmos de classificação. Na segunda etapa, um conjunto de algoritmos de classificação serão treinados com as categorias e parágrafos analisados pelos participantes. Por fim, na terceira etapa, um processo de validação será executado, de modo a encontrar o algoritmo de classificação que melhor identifique as categorias em documentações de projetos de software livre. Uma ilustração dos passos a serem executados é apresentada na Figura \ref{fig:metodo-classif}.

\subsubsection{Preparação dos dados}
A preparação dos dados consistirá em dividir o conteúdo das planilhas entre características e rótulos para classificação. Neste estudo, serão considerados como características os parágrafos das planilhas analisadas pelos alunos, e rótulos as categorias que eles identificaram para cada um dos parágrafos. Como algoritmos de classificação só são capazes de compreender valores numéricos, os parágrafos serão convertidos em valores estatísticos e heurísticos. Serão considerados como valores estatísticos, a transformação dos parágrafos em características TF-IDF obtidas com o uso de bibliotecas de aprendizado de máquina. Os valores heurísticos serão obtidos por meio de análise manual dos parágrafos e categorias marcadas. Quando um padrão linguístico que associe parágrafos a uma categoria for identificado pelo pesquisador, um novo valor heurístico será gerado como característica para classificação. 

A ideia de valores heurísticos emergiu do estudo realizado por \citet{prana2019categorizing} que identificou, por exemplo, que a categoria "cabeçalho da documentação" proposta em seu estudo poderia ser identificada pela ocorrência do título do repositório de código na documentação analisada. Caso as características obtidas a partir destas duas estratégias não sejam suficientes, outras estratégias poderão ser aplicadas nos dados obtidos a partir das planilhas. As marcações das categorias feitas pelos alunos serão convertidos em uma matriz binária de rótulos, seguindo a mesma estrutura disposta nas planilhas. Além da divisão entre rótulos e características para classificação, os valores convertidos também serão divididos em conjunto de treino, validação e teste, a serem utilizados nas etapas seguintes.

\subsubsection{Treinamento}
A etapa de treinamento será responsável pela construção do modelo de classificação que identificará categorias relevantes a novatos em documentações de software livre. Como foi estabelecido que mais de uma categoria poderá ocorrer para um mesmo trecho de documentação, classificadores multi-rótulos serão utilizados para treinamento dos modelos de classificação. Dado que a classificação multi-rótulos tende a ser mais complexa que outros tipos de aprendizagem, a estratégia de relevância binária será utilizada durante toda a etapa de treinamento. Na relevância binária, um problema de classificação com múltiplas classes como este, é transformado em problemas de classificação binária. Em outras palavras, o algoritmo de aprendizagem, ao invés de tentar identificar matematicamente todos as categorias de uma única vez nos trechos de documentação, tentará identifica-las separadamente, unindo os resultados ao final do processo. Esta é uma estratégia comumente utilizada por outros estudos que envolvam problemas multi-rótulos e classificação de textos como este. Durante o treinamento, o conjunto de treino será utilizado para alimentar os modelos de classificação.

Entre os algoritmos de aprendizado a serem treinados, se encontram: Random Forest (RF), Linear Support Vector Machine (SVM), Multinomial Naive Bayes (NB), K-Neighbors (kNN), Logistic Regression (LR), Multi-layer Perceptron (MLP), Linear Support Vector Machine + Stochastic Gradient Descent Learning (SGD). Estes algoritmos foram selecionados com base no que vem sendo utilizado por trabalhos relacionados, em especial, os estudos propostos por \citet{prana2019categorizing} e \citet{panichella2015how}. Além de trabalhos relacionados, os algoritmos MLP e SGD são recomendações propostas por desenvolvedores da Google\footnote{\url{developers.google.com/machine-learning/guides/text-classification}} e Scikit-learn\footnote{\url{scikit-learn.org/stable/tutorial/machine$\_$learning$\_$map}}, respectivamente.

\subsubsection{Validação}
Na etapa de validação, os modelos treinados serão avaliados a fim de se obter aquele que melhor identifique as categorias desejadas. Para isso, duas estrateǵias serão combinadas, validação cruzada e métricas de acurácia. Na estratégia de validação cruzada, diversos conjuntos de treino e validação serão gerados a partir dos dados das planilhas, e estes serão avaliados por um algoritmo de validação \textit{k-fold} a fim de compreender quais modelos de classificação gerados são capazes de generalizar o problema desejado. Além da validação cruzada, métricas de acurácia serão utilizadas, em especial, \textit{precision}, \textit{recall} e \textit{f-measure}, todas amplamente conhecidas e utilizadas por demais trabalhos envolvendo problemas de classificação. Estas métricas irão avaliar utilizando conjuntos de treino e validação pré-estabelecidos a acurácia do classificador. O modelo que obtiver os melhores resultados será submetido a etapa de avaliação (Seção \ref{sec:avaliacao}), e também será deste modelo em que as métricas de acurácia serão extraídas para apresentação dos resultados por meio do conjunto de teste.  

\subsection{Avaliação}
\label{sec:avaliacao}
Além da construção do modelo de classificação, duas análises serão entregues como parte resultante deste trabalho. Entre as atividades, serão entregues uma análise da importância das características do modelo de classificação para a predição das categorias, e uma avaliação das predições feitas pelo modelo, a serem avaliadas por novos e experientes contribuidores vindos de projetos de software livre. A descrição de cada uma das atividades é definida abaixo.

\subsubsection{Avaliação das características}
Assim que um modelo de classificação for selecionado após a etapa de validação, uma investigação  das características deste modelo será executada. A ideia é compreender quais características são importantes para identificação de cada uma das categorias relevantes a novatos em software livre. Tanto as características estatísticas como as heurísticas serão analisadas. Para que essa análise seja possível, os pesos de cada uma das características serão extraídos do modelo de classificação elegido, através da própria biblioteca de aprendizado de máquina a ser utilizada durante a construção do modelo. Quanto maior o peso de uma característica, consequentemente maior será sua importância para identificação de uma ou mais categorias. A relevância das principais características será apresentada por meio de gráficos de dispersão. 

Além disso, uma avaliação da diferença de importância entre características estatísticas e heurísticas também será efetuada. Para isso, dois novos modelos de classificação similares aquele selecionado na validação serão criados, um contendo apenas o conjunto de valores estatísticos, e outro apenas o conjunto de valores heurísticos. O valor de \textit{f-measure} será calculado para ambos os modelos. A ideia é compreender qual conjunto de valores tem maior impacto na identificação das categorias relevantes a novos contribuidores. O conjunto de características do modelo que obtiver o melhor valor \textit{f-measure}, será definido como aquele com o maior relevância para identificação das categorias.

\subsubsection{Avaliação das predições}

A última etapa deste método consistirá em avaliar se as predições feitas pelo modelo de classificação selecionado durante a validação fazem sentido para contribuidores no contexto de software livre. Para isso, novos e experientes contribuidores serão convidados, através de um questionário, a avaliarem trechos de documentação identificados pelo modelo de classificação. Documentações de um novo conjunto de projetos serão utilizadas durante esta etapa.
