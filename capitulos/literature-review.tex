\section{Software Livre}

\label{cap:free-software}
Software livre (do inglês, \textit{free software}) é o software que em sua essência é distribuído livremente, geralmente acompanhado por uma licença de software, e que conta com a disponibilização de seu código-fonte em algum meio acessível a usuários e desenvolvedores \citep{Gacek:2004:MMO:968720.968793}. Um programa de computador para ser considerado livre, deve fornecer a quem o utiliza quatro liberdades essenciais \citep{stallman2002free}:
\begin{itemize}
    \item A liberdade de executar o programa, para qualquer propósito;
    \item A liberdade de estudar como o programa funciona, e poder adaptá-lo;
    \item A liberdade de redistribuir cópias do programa a outros usuários;
    \item A liberdade de aperfeiçoar o programa, e poder liberar os aperfeiçoamentos a comunidade.
\end{itemize}
Outro termo geralmente associado a software livre é o conceito de software de código aberto (do inglês, \textit{open source software}). Apesar das diferenças\footnote{\url{https://www.gnu.org/philosophy/open-source-misses-the-point.html}} entre software livre e de código aberto, é possível considerar os termos como sinônimos. Um possível antônimo para software livre é o software proprietário, que restringe legalmente a terceiros o acesso ao código fonte do programa desenvolvido~\citep{fuggetta2003open}.

\subsection{Representatividade}
Embora o termo software livre ainda seja desconhecido para muitos usuários, projetos neste contexto tem mostrado grande influência no cenário de desenvolvimento e consumo de software. O servidor \textit{Apache} é um exemplo de projeto de software livre popularmente conhecido entre desenvolvedores ao redor do mundo. Mantido por voluntários, o \textit{Apache} hospedou em Julho de 2019\footnote{\url{w3techs.com/technologies/overview/web_server/all}}, mais de 44$\%$ dos \textit{websites} ativos na internet. O \textit{Mozilla Firefox} é outro exemplo de projeto de software livre que tem apresentado grande adesão de usuários no mercado de software, podendo ser considerado um dos maiores navegadores existentes, representou Agasto de 2019 uma parcela de mais de 5$\%$\footnote{\url{w3counter.com/globalstats.php}} na utilização de navegadores por todo o mundo.

Não apenas usuários e desenvolvedores tem demonstrado interesse em software livre, evidências mostram que diversas empresas também se encontram alinhadas a esta mesma filosofia de liberdade~\citep{pinto2018challenges}. A linguagem de programação Swift\footnote{\url{developer.apple.com/swift/blog/?id=34}}, desenvolvida pela Apple, é um exemplo de programa de computador que por muito tempo foi proprietário, e que hoje faz parte do contexto de software livre. Até mesmo a Microsoft, reconhecida por sua popularidade com projetos de código fechado como \textit{Windows} e \textit{Office}, lançou em Novembro de 2015 uma versão aberta do seu editor de texto, o \textit{Visual Studio Code}\footnote{\url{github.com/microsoft/vscode/issues/60}}. Estas companhias parecem não só estarem interessadas em tornar disponíveis publicamente o código fonte de seus projetos, como também tem mostrado interesse em contribuir com o cenário de software livre, pagando desenvolvedores para trabalharem em projetos neste contexto~\citep{pinto2018gets}.

\subsection{Funcionamento}
Geralmente, um novo projeto de software livre se inicia quando um programador possui uma ideia, problema ou porção de código a ser desenvolvido~\citep{raymond1999cathedral}. O programador submete uma proposta à comunidades colaborativas de software, anunciando seu projeto a outros desenvolvedores. Tais desenvolvedores, quando atraídos pelo projeto proposto, entram em contato com o programador, e passam a participar do desenvolvimento dos códigos. Com o tempo e com a adesão de novos contribuidores, uma nova comunidade de software livre se forma em torno do projeto. Plataformas de codificação passam a ser utilizadas como meio de gerenciamento do código fonte, e canais de comunicação são estabelecidos entre os membros da comunidade. 

Projetos neste contexto usualmente são mantidos a partir de contribuições de desenvolvedores internos e externos as comunidades de software livre. Para contribuir com um projeto, um desenvolvedor externo adquire uma cópia do código fonte através de uma plataforma de codificação, implementa modificações, e submete as mudanças novamente ao repositório de código do projeto~\citep{mockus2000case}. As modificações são revisadas por um grupo de mantenedores e, quando de acordo com os critérios de aceitação, são anexadas aos binários do programa. Desenvolvedores interessados em participar de uma comunidade de software livre, tendem a construir sua reputação por meio de contribuições ao código fonte dos programas desenvolvidos, de onde com o tempo, passam a ganhar espaço e notoriedade~\citep{von2003community}. O desenvolvimento dos projetos usualmente ocorre de maneira colaborativa, e as contribuições são feitas de forma voluntária, ou seja, desenvolvedores não são remunerados por suas colaborações~\citep{osterloh2007open}.

\subsection{Organização}
Em relação a organização dos projetos, \citet{nakakoji2002evolution} sintetizam a estrutura social de comunidades de software livre a partir de um modelo de camadas. Segundo os autores, nas camadas mais internas se encontram os principais desenvolvedores, aqueles que geralmente contribuem com a maior parte do código fonte e são responsáveis por revisar futuras contribuições. Ao meio da estrutura se encontram membros do projeto que são responsáveis por contribuir com pequenas correções e aprimoramentos. E por fim, nas camadas mais externas, se encontram diferentes níveis de usuários, que não contribuem com o código fonte do projeto, mas que eventualmente relatam problemas e dão sugestões, além é claro de desfrutar dos recursos oferecidos pelo programa. Além dos grupos apresentados pelo modelo de \citet{nakakoji2002evolution}, existem ainda os novos contribuidores que, apesar de não necessariamente constituírem parte da hierarquia de um projeto, por serem considerados essenciais ao desenvolvimento dos projetos, são amplamente estudados pela literatura em software livre. Neste contexto, um novo contribuidor, novato ou recém-chegado, pode ser definido como um desenvolvedor interessado em ingressar em uma comunidade de software livre por meio de contribuições. Entre as razões pelas quais novatos são amplamente estudados pela literatura, se encontram as dificuldades que estes enfrentam ao tentarem ingressar em projetos de software livre, como mostra a seção a seguir.

\section{Barreiras enfrentadas por novatos}
\label{cap:newcomers-barriers}
% Discuss Barriers in OSS
% Mention Igor Steinmacher and the categories he defined for newcomers barriers in OSS
Como as contribuições em software livre são em sua essência voluntárias, o ato de atrair novos contribuidores para tais comunidades é considerado fator essencial para continuidade de muitos projetos \citep{qureshi2011socialization}. Como em qualquer atividade coletiva, se membros ao saírem de uma comunidade não forem substituídos por novos contribuidores, a comunidade irá eventualmente deixar de existir, já que novatos podem ser considerados fonte de inovação, ideias e trabalho \citep{kraut2012challenges}. 

No entanto, atrair novos contribuidores para comunidades de software livre não é uma tarefa trivial \citep{zhou2015whowill}. Em um estudo sobre empresas que compartilham comunidades neste contexto, \citet{dahlander2008how} mostram que até mesmo para comunidades bem estabelecidas como o MySQL, diversos problemas são enfrentados ao atrair novos contribuidores. Entre os problemas que justificam a baixa adesão de novatos, estão barreiras enfrentadas por estes durante a entrada nos projetos de software livre. 

\citet{von2003community} descrevem tais barreiras como um conjunto de dificuldades, empecilhos ou deficiências no processo de contribuição, que dificultam com que  desenvolvedores realizem suas primeiras colaborações. De acordo com os autores, as barreiras estão intrinsecamente relacionadas ao nível de especialização de cada desenvolvedor. Como a complexidade de um projeto tende a crescer conforme seu desenvolvimento, ingressar em meio a sua execução tende a ser um processo lento e dificultoso a novos desenvolvedores que, por muitas vezes, desconhecem os processos e tecnologias utilizadas pela comunidade.

\citet{hannebauer2017relationship} dividem barreiras enfrentadas por novatos em duas categorias, submissão e modificação. De acordo com os autores, novatos tendem a enfrentar dificuldades tanto durante a elaboração de uma nova contribuição, como durante o processo de submissão do código modificado ao projeto. Em uma perspectiva complementar, \citet{steinmacher@thehard} mostram que tais barreiras não somente estão associadas as motivações e características do desenvolvedor ao contribuir com um projeto, como também são desencadeadas por problemas da própria comunidade a qual ele almeja participar. Problemas como falta de documentação e a má recepção fornecida a novos contribuidores, são exemplos de problemas que podem impactar a entrada dos novatos em comunidades de software livre.

Entre os estudos que sintetizam as dificuldades enfrentadas por novatos em software livre, se encontra o modelo de barreiras proposto por \citet{steinmacher@preliminary}. Dividido em seis categorias diferentes, o modelo apresenta cinquenta e oito barreiras enfrentadas por novos contribuidores. Entre as categorias, foram constatadas dificuldades relacionadas ao processo de recepção dos novatos, a características dos próprios desenvolvedores, a falta de orientação durante o processo de contribuição, a problemas com documentação, a problemas associados a diferenças culturais e a obstáculos técnicos. A Figura \ref{fig:barreiras} apresenta uma tradução livre do modelo de barreiras proposto pelos autores. Vale ressaltar, que tal modelo serviu de inspiração para elaboração de categorias relevantes a novatos em software livre utilizadas neste estudo, e apresentadas na Tabela \ref{tab:categories}.

O fato de tais comunidades não atraírem novos contribuidores o suficiente, usualmente implica na falta de condições necessárias para se manter o desenvolvimento dos projetos ativo, levando, por consequência, muitas comunidades de software livre a inatividade. Por esta razão, é tido como proposta deste estudo a elaboração de uma ferramenta que contribua com problemas que novatos enfrentam em relação as documentações dos projetos de software livre, de modo com que algumas das barreiras apresentadas por \citet{steinmacher@preliminary}, e demais trabalhos, sejam minimizadas ou deixem de existir. Na seção a seguir, uma visão mais minuciosa sobre problemas enfrentados em documentações de software é apresentada.

\section{Problemas em documentações de software}
\label{cap:documentation-problems}
A documentação de um programa de computador, quando correta, completa e consistente, tende a ser uma crucial fonte de informações para desenvolvedores \citep{kajo2005survey}. Isto porque
o propósito ao se documentar um programa, é garantir que todos os processos desenvolvidos durante sua elaboração sejam preservados \citep{aghajani2019issues}. Apesar dos custos de elaboração, a documentação de um sistema pode apresentar diversos benefícios ao seu desenvolvimento, tendo reflexo, por exemplo, na produtividade dos desenvolvedores, na redução dos custos de manutenção, e na extensão da vida útil do sistema desenvolvido \citep{zhi2015cost, deSouza2005maintenance}.

No entanto, evidências mostram que a documentação de programas de computador costumam apresentar problemas \citep{briand2003enough}. Em muitos casos, as documentações se encontram incorretas, obsoletas, incompletas ou incoerentes a realidade do sistema, o que pode comprometer o entendimento do programa por parte de desenvolvedores e mantenedores \citep{aghajani2019issues, forward2002relevance}. Problemas de documentação não se restringem a um único cenário ou área de aplicação, e podem estar presentes em diversos contextos do desenvolvimento de software.

No contexto de software livre, onde a documentação dos projetos é crucial para integração de novatos a comunidades, problemas também são enfrentados. Além de relatos associados a documentações obsoletas, incompletas ou incoerentes, \citet{steinmacher@preliminary} identificam eu seu modelo de barreiras outros tipos de problemas, tais como a falta de clareza nas informações dispostas em documentações, a dispersão de informações sobre diferentes meios, e a sobrecarga de informações.

\section{Trabalhos Relacionados}
\label{cap:related-work}
Nesta seção, são apresentados trabalhos que buscam solucionar problemas similares aos objetivos deste estudo, incluindo o apoio a entrada de novatos em software livre e a automatização da busca por trechos em documentações de software. Dois tipos de trabalhos relacionados são apresentados para cada tópico a seguir, trabalhos encontrados na literatura disponível, e trabalhos provenientes de fora do contexto acadêmico.

\label{cap:related-work}
\subsection{Apoio a novatos em software livre}

Na literatura, \citet{malheiros2012changes} propõem uma ferramenta denominada \textit{Mentor} que, através de algoritmos de recomendação, sugere a novos contribuidores arquivos do código fonte de projetos de software livre que podem contribuir com a realização de uma determinada tarefa. Tal ferramenta facilita a procura de arquivos que estejam associados a, por exemplo, a uma potencial solução de um erro no programa desenvolvido, de modo que tal solução pudesse ser desenvolvida por um novo contribuidor. Outra ferramenta similar é proposta por \citet{cubranic2005hipikat}, que também utiliza de algoritmos de recomendação, para sugerir a novos contribuidores arquivos do código fonte que possam contribuir na execução de uma tarefa. A diferença é que, nesta ferramenta, não são sugeridos apenas arquivos de código fonte, mas também trechos provenientes da documentação do projeto, de canais de comunicação, de relatórios de erro e de planos para teste, que possam ser relevantes para solução da tarefa a ser desenvolvida pelo contribuidor.  

Buscando auxiliar novatos na escolha de uma primeira tarefa, \citet{wang2011which} propõem um mecanismo inteligente de busca por erros não resolvidos em projetos de software livre. De acordo com os autores, uma boa maneira de ingressar em um projeto de software livre se encontra na resolução de erros do programa desenvolvido.  Encontrar erros que se adéquem as características técnicas e de interesse do novo contribuidor, é um fator que deve ser levado em consideração, já que pode ser primordial na integração do novato a projetos neste contexto. O sistema proposto tem acesso a lista de erros de um projeto, e permite que desenvolvedores encontrem erros que desejam solucionar por meio de buscas inteligentes, além de identificar erros similares aqueles que os novatos demonstrem interesse.

Engajados com a orientação de novatos em software livre, \citet{canfora2012going} descrevem uma abordagem de auxílio a novatos denominada \textit{Yoda}, que objetiva identificar e recomendar mentores em projetos de software livre para novos contribuidores. Por meio de algoritmos de recomendação, e utilizando dados extraídos de listas de e-mail e sistemas de controle de versão, o \textit{Yoda} recomenda a todo novato que se interessa por um projeto de software livre, um mentor comprometido a supervisionar a sua contribuição a comunidade que ele pertence. A recomendação de um mentor ocorre através de uma análise sistemática do histórico de contribuição e mentoria de cada desenvolvedor interno em um determinado projeto, que é associado pelo próprio sistema a um novato interessar em contribuir com a comunidade.

Todas estas ferramentas descritas acima tem relação com o modelo de barreiras proposto por \citet{steinmacher@preliminary}, que também propõem como contribuição de seu artigo, uma ferramenta para auxiliar novatos em software livre. Conhecida como \textit{FLOSSCoach}, tal ferramenta é uma plataforma de auxílio a novos contribuidores, que visa orientar novatos entre os principais caminhos que devem ser seguidos para realização de uma contribuição de sucesso em projetos de software livre. Tais caminhos foram definidos a partir das principais dificuldades que novatos enfrentam em neste contexto, como problemas em compreender o fluxo de contribuição de um projeto, problemas com a submissão de contribuições e para manter contato com as comunidades. 

Além de trabalhos na literatura, uma diversidade de ferramentas também são propostas por desenvolvedores fora do contexto acadêmico. O \textit{Up For Grabs}\footnote{http://up-for-grabs.net}, por exemplo, é um website que tem como objetivo auxiliar novatos a encontrarem tarefas disponíveis em projetos de software livre. Os novatos acessam o website que, apresenta a eles, uma série de projetos que contenham tarefas nos sistemas de caça tarefas (do inglês, \textit{issue tracker}) específicas para novos contribuidores, como mostra a Figura \ref{fig:up-for-grabs}. Outro exemplo de ferramenta é o \textit{First Timers Only}\footnote{http://www.firsttimersonly.com/}, um website que estabelece diretrizes para novatos que desejam contribuir com projetos de software livre. Nesta página é possível encontrar uma diversidade de dicas e tutoriais relacionados aos primeiros passos de um novato em projetos neste contexto. Entre as páginas apresentadas pelo website, se encontra o próprio \textit{Up For Grabs}.

\subsection{Ferramentas relacionadas a documentações de software}
Quanto a classificação de documentações, \citet{prana2019categorizing} elaboraram um modelo de classificação que identifica as principais seções existentes em arquivos de documentação README, usualmente presentes em repositórios de código aberto. Os autores identificaram manualmente, oito categorias usualmente presentes em arquivos README, e construíram um modelo de identificação que fosse capaz de identificar tais categorias em arquivos de demais projetos.
